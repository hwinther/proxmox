hjemmelab

En liten klynge som driftes som et lite datasenter — alt beskrevet i Git, og som retter seg selv. Bygget på Proxmox, k0s, Cilium og Flux.

~33 kjørende tjenester 2 klynger (prod + edge) 100% GitOps
01 · Problemet

Hvorfor i det hele tatt orkestrering?

Slik vi gjorde det før

  • Docker-containere startet for hånd på én og én vert
  • IIS-app-pools og Windows-tjenester satt opp via RDP og klikk
  • Krasjer noe? Noen må logge inn og starte det på nytt
  • «Det funker på min server» — config sklir fra hverandre

Med Kubernetes

  • Du beskriver ønsket tilstand — klyngen sørger for resten
  • Krasjer noe, startes det automatisk på en frisk node
  • Skalering, plassering og helsesjekk er innebygd
  • Én kilde til sannhet, ingen manuell drift

03 · Den store ideen

Deklarativt — klyngen speiler Git

I stedet for å fortelle serveren hvordan noe skal gjøres, beskriver vi hva resultatet skal være. En agent i klyngen passer kontinuerlig på at virkeligheten matcher beskrivelsen.

  • Push til Git → endringen ruller ut av seg selv
  • Noen endrer noe manuelt? → klyngen retter det tilbake
  • Full historikk → hver endring er en commit, med code review og rollback
💡

For Windows-folk: tenk Group Policy som aldri slutter å håndheve — bare for hele applikasjonsstacken, og versjonert i Git.

04 · Fundamentet

Plattformen — et datasenter på noen få bokser

  • Proxmox — virtualisering i bunn (VM / LXC)
  • k0s — lettvekts Kubernetes: 3-node HA control plane + workers
  • 2× Raspberry Pi — ARM-edge-noder for radio/SDR
  • Ceph — delt, replikert lagring (ceph-rbd)
Noen få fysiske bokser — én logisk klynge.
05 · Leveranse

GitOps med Flux

Flux er agenten som speiler Git inn i klyngen.

  • Push til Git → endringen ruller ut av seg selv
  • Én repo, to klynger — produksjon og edge-SDR
  • Avhengighetsrekkefølge — databaser før appene som bruker dem
≈ en release-pipeline som aldri slutter å sjekke at alt stemmer.
06 · Nettverk

Nettverk med Cilium (eBPF)

  • eBPF i kjernen — erstatter kube-proxy
  • Hubble — sanntidskart over hvem som snakker med hvem
  • Egress-gateway — fast utgående IP ut av klyngen
  • NetworkPolicy — egen «brannmur» per app
≈ software-definert nettverk + Wireshark-innsyn, innebygd.
07 · Sikkerhet

Sikkerhet som kode

  • Default-deny nettverkspolicy — hver app får sin egen «brannmur»; ingenting snakker sammen uten lov
  • Kyverno admission-regler — ingen :latest, kjør ikke som root, krev ressursgrenser, kun signerte images
  • Kubescape — løpende skann mot CIS-benchmarks og sikkerhetsstatus
  • OIDC / Authelia — single sign-on foran tjenestene
🛡️

Som Group Policy + brannmurregler + code review — men håndhevet automatisk ved hver eneste utrulling, ikke ved revisjon i etterkant.

08 · Innsyn

Observability — vi ser alt

  • Prometheus + Grafana — metrikker og dashboards (Ceph, Postgres, Proxmox, ESP-sensorer …)
  • Loki — sentraliserte logger fra alle tjenester
  • Tempo — distribuert tracing på tvers av kall
  • Alertmanager → Telegram — varsel rett på mobilen når noe ryker
📊

Tenk PerfMon + Event Viewer + APM — samlet på ett sted, med historikk og varsling i stedet for én server om gangen.

09 · Tilstand

Tilstandsfulle tjenester — databaser som drifter seg selv

  • CloudNativePG — PostgreSQL med automatisk failover og replikering, som en «managed MSSQL»-oppsett
  • RabbitMQ — meldingskø for tjenester som snakker asynkront
  • Valkey / Redis — cache og sesjonslagring
🗄️

For MSSQL-folk: dette er Always On-tankegangen — men operatøren håndterer oppsett, failover og oppgradering for deg.

10 · Sikkerhetskopi

Backup med Proxmox Backup Server

  • Planlagte jobber per app — hver tjeneste sikkerhetskopieres for seg
  • Kryptert — nøklene distribueres automatisk til riktig namespace
  • Gjenoppretting på forespørsel — egne restore-jobber klare når det trengs
💾

Som planlagte SQL-backups + VSS — men per app, kryptert, og selve oppsettet er versjonert i Git sammen med alt annet.
